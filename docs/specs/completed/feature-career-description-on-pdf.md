## §1 Feature Description

### Overview

Persist a structured `CareerDescription` (summary + 4–6 task bullets) on every `Build`, generated by Gemma from O*NET data, and render it as a new "About this career" section on page 1 of the My Build PDF. The same payload powers the per-career sparkle (✦) panel on `/set-your-course` so the description the student saw is the description that prints.

### Problem Statement

Today the per-career sparkle on `/set-your-course` opens `GemmaChat` with `scope.kind="career"`. Gemma describes the career, the student reads it, and on panel close the response is **discarded**. None of it reaches `Build`, none of it reaches the PDF.

Meanwhile the 2-page My Build PDF (`backend/app/services/pdf_export.py`) renders verdict, pentagon, cost/ROI, risk profile, skills, and audience questions — but **never tells the reader what the job is**. A parent skimming the PDF sees a 0–10 score for "Earnings ceiling" before they ever read what the career involves day-to-day.

This spec fills both gaps with a single artifact: a structured `{summary, tasks[]}` payload generated once per SOC, cached, and persisted on the `Build` so it is identical on screen and on paper.

### Success Criteria

- [x] On `/set-your-course`, clicking the per-career sparkle ✦ for SOC X renders a structured "About this career" header card in the panel within ~3s on a cold cache (summary + bulleted tasks). Reopen for the same SOC → instant. _(Implemented via `loadCareerDescription` single-flight store + `renderCareerDescriptionCard` skeleton/populated/error states in `GemmaChat`. Resolved via verified frontend tests; cache-hit behavior asserted in `sparkle_click_cache_hit_skips_fetch`.)_
- [x] Specing a build with that career produces a `Build` whose `career_description` field is populated with the **same** summary and tasks. _(Eager fetch joins `_gemma_fanout`'s `asyncio.gather` at `create_build`, `_build_stream`, and `rebuild_with_sliders`. Single-flight cache guarantees the same payload seen on screen prints to PDF. Asserted in `test_spawn_eager_career_description_attaches`.)_
- [x] Exporting the My Build PDF for that build renders an "About this career" section on page 1, between the header strip and the pentagon, showing the summary paragraph and a bulleted list of the same 4–6 tasks. _(Implemented in `pdf_export._about_this_career_section`. Inserted between verdict line and pentagon per architect-resolved placement. Asserted in `test_pdf_renders_about_this_career_section_when_present`.)_
- [x] Specing a build for a SOC the student **never** sparkle-clicked still produces a populated `career_description` (eager spawn-time generation). _(Build pipeline always invokes `career_description.get_or_generate` regardless of upstream sparkle clicks.)_
- [x] If the eager generation fails (timeout / Gemma unreachable / parse failure after retry), the build still succeeds with `career_description=None`. ## §2 Design Decisions

### Decision Log

| # | Decision | Rationale | Alternatives Considered |
|---|----------|-----------|------------------------|
| 1 | One structured Gemma call returns `{summary, tasks[]}`; same payload renders in sparkle panel and PDF | Guarantees the student-visible text equals the printed text — the user's hard requirement. Eliminates dual-source-of-truth drift. | (a) Snapshot the first message of the freeform chat reply: variable length, no copy guarantees, often tangents into Q&A. (b) Run two separate calls (chat + PDF): cannot drift fix without locking student out of seeing the PDF copy live. |
| 2 | Eager fetch on Spec My Build, with PDF-export lazy fallback | Most builds get the description pre-baked (instant export, available downstream). Older builds and recovery cases have a deterministic fallback. | (a) PDF-only lazy: cheaper tokens but adds latency to every export, description never available on-screen. (b) Eager only: brittle — older serialized builds and any eager-fail case would miss the section forever. |
| 3 | Backend pre-fetches `top_5_activities` from `get_task_breakdown`, then prompts Gemma to rewrite — Gemma does not call the MCP tool itself | Higher fidelity to source data; eliminates a class of "Gemma chose not to call the tool" failures; faster (one round trip instead of N). | Gemma tool-calling for tasks: matches the existing `ask_gemma.py` pattern but introduces non-determinism in what tasks come back. |
| 4 | Per-SOC in-process cache (`dict[str, CareerDescription]`) keyed by SOC alone | Same SOC → same description guaranteed across sparkle clicks and spawn-time fetches within a process. SOC is the right grain — career description is occupation-level, not student-level. | (a) Per-build cache: would regenerate for every student even within a session. (b) DuckDB-persisted cache: gold zone is sacred, this is generated content; over-engineering for hackathon scope. (c) SOC + locale: locale is currently hardcoded to `en` for this surface; revisit when localization arrives. |
| 5 | New endpoint at `GET /careers/{soc_code}/description` (not `/career/...`) | Aligns with existing namespace in `backend/app/routers/careers.py` (`/careers/{soc_code}/schools`). Singular path was a typo in the plan. | `/career/{soc_code}/description` (rejected — inconsistent with sibling routes). |
| 6 | New "About this career" section sits on **page 1**, after header strip, before pentagon | Reader hits "what is this job" before any quantitative score. Matches a parent's actual reading order. | (a) Page 2 top: doesn't disturb page 1, but lets the reader hit stats/cost/risk before they know the job. (b) Below risk profile on page 1: relegates description to supporting context — wrong priority. |
| 7 | Strict JSON-only parse with one retry; voice-validation with one retry; then `CareerDescriptionUnavailable` | Bounded latency budget; deterministic failure mode; no infinite-retry pathologies on a Gemma model that drifts. | Looser markdown / heuristic parsing: invites copy quality regressions and silent voice-guide violations. |
| 8 | Eager-fetch task uses `asyncio.create_task` with an 8s wall timeout, run in parallel with other spawn work | Time budget worst-case ≈ existing spawn p95 latency; failure does not block the build. | Synchronous serial: pessimizes spawn latency unnecessarily. |
| 9 | Lazy-fetch path persists the result back via `builds_service.save_build` | Future exports of the same build skip the call and ship identical text. | Render-only without persist: re-pays the Gemma cost on every export and risks drift if the model changes. |
| 10 | `career_description: CareerDescription \| None` on `Build`; nullable | Older serialized builds load cleanly. Eager-fail case is representable. PDF render branches on presence. | Required field with a sentinel value: makes serialization brittle, hides failure modes. |
| 11 | Voice owner: `@fp-copywriter` authors the system prompt; lives next to the service module as a constant | Matches the project pattern (Gemma's Take in `guidance.py`, boss narratives in `boss_fights.py`). Single owner for runtime narrative surfaces. | Inline in the service: makes copy iteration require an engineer; violates the fp-copywriter agent's stated remit. |
| 12 | **Tiered anchor fallback** (A→B→C→D) replaces binary "raise on null `top_5_activities`". Tier A=full activities; Tier B=BLS `description` only (24 partial-tier SOCs); Tier C=`occupation_title` + major-group only (33 SOCs missing from `onet_work_profiles`); Tier D=raise. Tier B/C render with explicit "AI-inferred from occupation summary/title" disclaimer chip (panel) + italic line (PDF). `CareerDescription.anchor_tier: Literal["activities","description_only","title_only"]` carries the tier. | Per @fp-data-reviewer: 7% of student paths hit empty `top_5_activities` — paramedics, EMTs, financial analysts, special-ed teachers, "all other" residual SOCs. Binary raise would silently delete the section from PDFs of high-emotional-stakes careers, which is the exact failure mode this spec exists to fix. Project rule: "no path is out of scope." | (a) Binary raise (per @fp-architect item 7): rejected — regresses 7% of paths. (b) Generate without disclaimer: rejected — violates project bright line on observed-vs-AI-estimated data. |
| 13 | **Single-flight cache:** `dict[str, asyncio.Future[CareerDescription]]` keyed by `(soc_code, _PROMPT_VERSION)` rather than `dict[str, CareerDescription]`. First miss creates and caches the Future; concurrent misses await the same Future. | Per @fp-architect item 5/7: a plain `dict` check-then-set under asyncio is not atomic across `await` boundaries. Under cloud-demo concurrency (parents+students sparkle-clicking the same hot SOC), this can blow OpenRouter rate limits. Versioning the prompt prevents staleness across copywriter iterations. | (a) Plain dict: brittle under concurrency. (b) `asyncio.Lock` per SOC: equivalent semantics, more LOC. |
| 14 | **Lazy persist via `state.update_build(build_id, build)`** (not `builds_service.save_build(build)` directly). Writes propagate through the cache + DuckDB in one canonical path. Race between two concurrent PDF exports is benign — single-flight cache returns identical content; `INSERT OR REPLACE` is last-write-wins on identical payload. | Per @fp-architect item 4: `_load_build_or_404` returns the cached instance by reference today, so direct `save_build` happens to stay coherent — but only by implementation detail. `state.update_build` is the codebase's canonical single-write path. | Direct `save_build`: fragile coupling, breaks if `_load_build_or_404` is refactored to deep-copy. |
| 15 | **Eager fetch lives in `backend/app/routers/builds.py`, joining `_gemma_fanout`'s existing `asyncio.gather`** at all three spawn sites (`create_build`, `_build_stream`, `rebuild_with_sliders`). Uses `return_exceptions=True`; failure → `career_description=None`. | Per @fp-architect item 1: `services/builds.py` is sync persistence only — no async orchestration. Streaming endpoint (`/builds/stream`, the canonical path the frontend uses) would silently miss the eager fetch if integrated at the wrong layer. Fanout shares one budget, one logging hook. | Separate `wait_for(timeout=8.0)`: pessimizes spawn latency; competes with fanout's semaphore independently. |
| 16 | **Voice validator promoted, not duplicated.** `pdf_questions._has_forbidden_term` → public `pdf_copy.contains_forbidden_term(text, terms: frozenset) -> bool`. `pdf_questions.py` re-imports it. `career_description.py` calls it with `RPG_TERMS_FORBIDDEN_IN_PDF`. | Per @fp-architect item 6: today two modules already redundantly compile the same validator; a third copy here would make drift inevitable. | Separate `voice_validation.py` module: over-engineering for hackathon scope; existing `pdf_copy.py` is the natural voice authority. |
| 17 | **MCP tool extension:** add `description` to `TASK_BREAKDOWN_RESPONSE_FIELDS` in `src/mcp_server/futureproof_server.py`. Required for Tier B (description-only anchor). One-line change to existing tool, no new tool. | Per @fp-data-reviewer: today neither `get_task_breakdown` nor `get_occupation_data` returns the BLS `description`; without this, Tier B is impossible and 24 partial-tier SOCs (cardiologists, paramedics, EMTs) silently regress. | (a) Direct DuckDB read from backend: rejected — creates dual-source-of-truth that decision #1 forbids. (b) Skip Tier B: rejected — re-introduces the failure mode this spec exists to fix. |
| 18 | **Retry decision tree by failure class.** Empty string from `gemma_client.generate()` → transport failure (1 retry, original prompt). Non-empty + parse fail → parse failure (1 retry, prompt + "Output **only** valid JSON matching the schema."). Non-empty + parse ok + voice fail → voice failure (1 retry, prompt + "Do not use the words: <list>."). | Per @fp-architect item 2: `gemma_client.generate()` returns `""` on failure, doesn't raise. Blind retries on the same prompt almost always reproduce the same drift; strengthened retry prompts have higher recovery rates. | Single fused retry budget: invites double-billing on parse-fail-after-empty masquerade. |

### Constraints

- **Bounded latency budget:** eager spawn fetch ≤ 8s wall; lazy PDF fetch ≤ 12s wall. Failure is non-fatal in both paths.
- **Cloud demo concurrency:** OpenRouter rate limits — the per-SOC cache is the rate-limit guard; do not bypass it.
- **Voice rules (PDF):** must pass `pdf_copy.RPG_TERMS_FORBIDDEN_IN_PDF`. Summary ≤ 500 chars; each task ≤ 90 chars; 4–6 task bullets.
- **No schema changes** to gold zone. Iceberg is untouched. This is a backend service + serialized Build field + PDF render; no DuckDB writes other than the Build store's existing persistence.
- **Locale:** scope is `en` only for this round. Locale-aware regeneration is explicitly out of scope.
- **Out of Scope:**
  - Surfacing `CareerDescription` on `BuildResultsScreen` or `FutureScreen` (data lives on Build; cheap follow-up later).
  - Pre-warming the cache for popular SOCs at server startup.
  - Persisting the SOC cache to disk across restarts.
  - Localization of the description to non-`en` locales.
  - Changing the freeform chat behavior of `GemmaChat` for the career scope — chat continues as today below the new header card.
  - Editing the description after generation (no admin override surface).

## §3 UI/UX Design

> @fp-design-visionary fills this section BEFORE implementation begins. Two surfaces in scope: (a) sparkle panel structured header card on `/set-your-course`, (b) "About this career" section on page 1 of the My Build PDF. Reference Brightpath tokens by name (see `DESIGN.md`). Do not hardcode pixel values — use existing tokens from `pdf_export.py` for typography/spacing.

### Mockups

**(a) Sparkle panel header card (`GemmaChat`, `scope.kind="career"`)**

**Emotional target:** *"Oh — this is who I become."* The student has just clicked the ✦ sparkle on a career card. The panel slides in (existing animation). What greets them at the top is not a loading dot and a chat prompt — it's a structured, settled card that reads like a quiet pre-brief from a knowing coach. The freeform chat is still there underneath, ready when they want to dig further. The header card is the bedrock; the chat is the conversation around it.

**Anatomy.** A bordered card flowering one elevation tier above the panel surface — `bg-bp-surface` (#2D3060) on top of the panel's `bg-bp-mid` (#232545) — so it reads as the **anchor block**, distinct from the chat stream below. The card slots into the panel directly under the existing `<header>` (the "Ask Gemma" + scope chip strip), above the scrollable chat feed. It does **not** replace any existing chrome; the header strip stays, the scope chip stays, the chat feed is just shorter on first paint.

**Surface placement in `GemmaChat.tsx`:** insert the card as the first child of the scrollable feed div (`ref={scrollRef}`), conditionally rendered when `scope?.kind === "career"` and `careerDescription` (or its loading/error sentinel) is provided via prop. It scrolls *with* the chat history — not pinned — so on a long conversation the student can scroll back up to re-read the brief. This is intentional: the description is part of the conversation, not chrome on top of it.

**State 3 — Loading (skeleton):**

## §4 Technical Specification

### Architecture Overview

A new `career_description` service owns the `{summary, tasks[], anchor_tier}` generation. It pre-fetches O*NET activities + BLS occupation `description` via the existing `mcp_client` wrapper for `get_task_breakdown` (with `description` newly exposed by an additive MCP tool change), then makes a single non-streaming `gemma_client.generate()` call with a strict JSON output contract. The chosen prompt variant depends on which anchor data is available (Tier A/B/C); Tier D raises before invoking Gemma. Results are validated against PDF voice rules. Cache is single-flight per `(soc, prompt_version)`.

The service is invoked from three sites:
1. **HTTP** — `GET /careers/{soc_code}/description` (new endpoint on `backend/app/routers/careers.py`); the frontend calls this on sparkle click.
2. **Eager spawn** — `backend/app/routers/builds.py` (the actual async spawn pipeline lives here, not `services/builds.py`). The eager fetch joins `_gemma_fanout`'s existing `asyncio.gather` at all three spawn sites: `create_build`, `_build_stream`, `rebuild_with_sliders`. Result attaches to `Build.career_description`. Failure → `career_description=None`; build still succeeds.
3. **Lazy PDF fallback** — `backend/app/routers/pdf_export.py`, before render: if `build.career_description is None`, fetch within a 12s `wait_for`, persist via `state.update_build(build_id, build)` (the canonical single-write path), swallow `CareerDescriptionUnavailable` / `TimeoutError`.

The PDF render layer reads `build.career_description` and emits a new "About this career" section on page 1, after the verdict line and before the pentagon. Tier B/C generations include a one-line italic disclaimer. Frontend stores the fetched description in a small SOC-keyed cache for instant reopens.

### Gemma Call Site Discipline

This spec adds **one new Gemma call site** at `career_description.get_or_generate()`. Required:

- **`gemma_client.generate()` returns empty string on failure, NOT raises.** The retry decision tree distinguishes empty-string (transport) from non-empty-but-malformed (parse) from non-empty-but-voice-violating. See Service Changes above.
- **Fallback behavior:** Tier A→B→C→D fallback covers anchor data shape; on Tier D or exhausted retries, the service raises `CareerDescriptionUnavailable`. Both call sites (eager + lazy) catch this and proceed without the description. The PDF, sparkle panel, and Build all tolerate `None`.
- **Logging:** every call writes to `logs/gemma.jsonl` per existing `gemma_client.generate()` instrumentation. No additional logging needed — confirm during implementation that the existing instrumentation covers this path.
- **Both backends:** works under `INFERENCE_BACKEND=ollama` (local) and `INFERENCE_BACKEND=openrouter` (cloud demo) — `gemma_client.generate()` already abstracts the backend.
- **Rate limits / concurrency (OpenRouter cloud):** the single-flight `dict[tuple[str, str], asyncio.Future]` cache is the primary guard. Concurrent sparkle clicks for the same hot SOC await the same Future. The two-call retry budget caps a single (soc, prompt_version) worst case at 2 Gemma round trips.
- **Existing call sites untouched:** this spec does not modify `guidance.py`, `boss_fights.py`, `next_steps.py`, `skill_recs.py`, or `ask_gemma.py`. Their fallbacks and behavior are unchanged.

## §5 Architecture Review

### @fp-architect Review
**Status:** CHANGES REQUESTED
**Reviewed:** 2026-05-07

#### System Context

This feature adds one Gemma narrative surface on top of the existing Bronze→Silver→Gold→MCP stack. It does **not** introduce any zone changes: O*NET data already lives in Gold (`consumable.onet_work_profiles`) and is read through the established `mcp_client.call("get_task_breakdown", ...)` path. The new `career_description` service is a thin Gemma-bound transformer that sits between the Gold zone (read) and three call sites: a new HTTP endpoint, the build-spawn pipeline, and the PDF export router. The artifact (`CareerDescription`) crosses the FastAPI boundary as a Pydantic v2 model and is persisted on the existing `Build` row's `data` JSON blob — no new DuckDB columns. Architecturally clean in shape; the issues below are about contract correctness inside that shape.

#### Data Flow Analysis

```
Sparkle click  -->  GET /careers/{soc}/description  -->  career_description.get_or_generate(soc, title)
                                                          |
Spawn pipeline -->  routers/builds.py:create_build      --+--> per-SOC cache hit? --> return
                    routers/builds.py:_build_stream       |        miss
                    routers/builds.py:rebuild_with_sliders|         |
                                                          |         v
PDF export    -->  routers/pdf_export.py (lazy fallback)--+    mcp_client.call("get_task_breakdown")
                                                          |    mcp_client.call("get_occupation_data")
                                                          |         |
                                                          |         v
                                                          |    gemma_client.generate(...)  [empty-string-on-failure]
                                                          |         |
                                                          |         v
                                                          |    strict json.loads + voice validate (1 retry each)
                                                          |         |
                                                          |         v
                                                          +--> CareerDescription | CareerDescriptionUnavailable
```

Boundary crossings: (1) HTTP request → service (typed via Path pattern + Pydantic response_model); (2) service → MCP (typed via mcp_client schema validation); (3) service → gemma_client (untyped string in/out, **empty-string-on-failure contract**); (4) service → caller (Pydantic CareerDescription or named exception); (5) Build → DuckDB (`builds.data` JSON blob, model_dump_json round-trip); (6) Build → PDF render (in-process attribute access).

The data flow is sound. The hops that need contract tightening are (3) and the Build mutation/persist hop on the lazy path — see Concerns.

#### Contract Review

`CareerDescription` Pydantic model is well-shaped: SOC pattern matches the project-wide `_SOC_PATTERN`, summary/task length caps are enforced at validation time, generated_at and model fields support audit. The `Build.career_description: CareerDescription | None = None` evolution is backwards-compatible: Pydantic v2 applies the default during `Build.model_validate_json` on older serialized rows, and the existing DuckDB schema stores Build as a JSON blob (`builds.data`) so no ALTER TABLE is needed. `BuildSummary` does not include the new field — correct, since the description is heavyweight and not part of list views.

The new `GET /careers/{soc_code}/description` endpoint matches the namespace convention in `routers/careers.py` (sibling: `GET /careers/{soc_code}/schools`, same `_SOC_PATTERN` Path validator, same 502/422 error mapping pattern). Use the same `_dispatch`-style error wrapper or replicate it inline.

#### Findings

##### Sound
- Decision 1 (single structured payload powers both surfaces): eliminates the screen-vs-print drift class of bugs at the architecture level. This is the right call.
- Decision 3 (backend pre-fetches activities, Gemma rewrites): correct for this surface. ConceptNormalizer-style tool-calling is overkill for a deterministic occupation-level transform. The `mcp_client.call` path is sync but already wrapped in `asyncio.to_thread` patterns elsewhere in the codebase.
- Decision 10 (nullable field): correct for backwards compatibility with all existing serialized Builds.
- Endpoint namespace + SOC pattern reuse on the new route: consistent with `routers/careers.py`.
- Persisting on the existing JSON blob path: respects the rule "no schema changes without a spec" — the field rides inside `data` and is opaque to the columnar projection used by `BuildSummary`/`list_builds`.

##### Concerns

- **Wrong file for the eager-spawn integration point.** `backend/app/services/builds.py` only contains the sync factory `build_from_parts(...)` and the DuckDB persistence layer — no async orchestration. The actual concurrent build assembly lives in `backend/app/routers/builds.py` in three places: `create_build` (POST `/builds`), `_build_stream` (SSE POST `/builds/stream`), and `rebuild_with_sliders` (POST `/builds/{id}/rebuild`). All three call `build_from_parts(...)` after `_gemma_fanout`. **Impact:** If the implementer reads the spec literally and adds the eager fetch to `services/builds.py`, the streaming endpoint (`/builds/stream` — the canonical path the frontend uses) will silently miss the eager fetch entirely, every spawn will fall through to the lazy PDF path, and demo p95 export latency will balloon by 12s on the cold path. **Recommendation:** Update §4 File Changes to target `backend/app/routers/builds.py` and explicitly enumerate the three integration sites. The eager fetch should join the `_gemma_fanout` `asyncio.gather(...)` (or run alongside it) so the description shares the same time budget as the rest of the Gemma fan-out, not a separate 8s wall on top of it. For `_build_stream`, the description should be either yielded as its own SSE event (`event: career_description`) or attached to the final-build commit in the existing `as_completed` loop — not the skeleton.

- **`gemma_client.generate()` returns empty string on failure, not an exception.** The spec's pseudocode treats Gemma errors as raising. They don't. **Impact:** A network-down or transport-timeout case will silently produce `raw == ""`, the strict `json.loads` path returns None / raises `JSONDecodeError`, the retry runs and may also return "" — net effect is two wasted Gemma calls before `CareerDescriptionUnavailable`. Worse: code that does `if not raw: retry` could mask a parse-failure-after-real-response as "empty response, retry" and double-bill. **Recommendation:** §4 Service Changes must document this contract. Define the retry decision tree explicitly: empty string ⇒ transport failure (count as 1 failure, retry once); non-empty + parse fail ⇒ parse failure (retry once with stricter prompt); non-empty + parse ok + voice fail ⇒ voice failure (retry once with explicit "do not use these words" reminder). Two separate retry budgets is a cleaner mental model than one fused budget. Match the `pdf_questions.py` pattern: it returns `None` from `_parse_response` and routes through a `gemma_path="fallback_malformed"` enum — reuse that idiom.

- **`asyncio.wait_for` already cancels on timeout — `desc_task.cancel()` after `TimeoutError` is a no-op (and the line in §4 is misleading).** Per Python docs, `asyncio.wait_for(task, timeout)` cancels the inner task when the timeout fires before re-raising `TimeoutError`. **Impact:** Cosmetic bug in the spec — won't break anything but signals weak async hygiene if reviewers/implementers cargo-cult it. **Recommendation:** Drop the `desc_task.cancel()` line. Replace with a comment: `# wait_for already cancelled the task on TimeoutError per asyncio contract`. The catch should be a single `except (asyncio.TimeoutError, CareerDescriptionUnavailable) as exc:` with no cancel call.

- **Lazy persist must update `state._builds` cache or subsequent reads see stale data.** The PDF endpoint resolves Build via `_load_build_or_404` → `state.get_build` → `_builds[build_id]`. The lazy path mutates `build.career_description` in place and calls `builds_service.save_build(build)`, which writes DuckDB but does **not** touch the in-memory state cache. Because the mutation is in-place on the cached object, the cache happens to stay coherent here — but only because of an implementation detail (Python passes by reference and `_load_build_or_404` returns the cached instance). If anyone refactors `_load_build_or_404` to deep-copy or model_validate before returning, this silently breaks. **Impact:** Fragile coupling between cache and persist. **Recommendation:** Use `state.update_build(build_id, build)` (already exists in `backend/app/state.py:44` and calls `save_build` itself) instead of `builds_service.save_build(build)`. This makes the cache update explicit and idempotent — and `state.update_build` is the canonical single-write path the codebase already uses for build mutations.

- **Lazy persist race condition between two concurrent PDF exports.** If two PDF requests for the same `build_id` arrive within the 12s eager window (refresh, double-click, parent + student exporting simultaneously), both see `career_description is None`, both fire `get_or_generate`, both await ~3-12s, both call `save_build`. The per-SOC in-process cache **does** dedupe the Gemma call (good — same SOC, second await hits the populated cache). But the writes still race: `INSERT OR REPLACE` is last-write-wins on `(build_id, ...)`, and the two writes will produce identical content (cache hit means same payload) so the race is **benign for content** but doubles the DuckDB write pressure. **Impact:** Low — content correctness preserved, write amplification is 2×. **Recommendation:** Document this as accepted in §2 Decision Log as a new row. If concerned, gate the lazy fetch on a per-build `asyncio.Lock` (cheap, in-process). Not blocking.

- **Per-SOC cache is unbounded and unbounded across process lifetime.** `dict[str, CareerDescription]` with no LRU, no max size, no TTL. SOC space is ~800 codes; payload is ~600 chars + tasks. Worst case ~500KB. Fine for a long-running process. But: cache invalidation on prompt-edit is impossible without restart, and there's no eviction story for "the demo found a regression in the prompt mid-presentation." **Impact:** Operational, not architectural. **Recommendation:** Acceptable as-is for hackathon scope. Add a one-line `clear_cache()` helper for tests and emergency purposes (matches the `gemma_client.reset_cache()` pattern). Note in Decision 4: "no eviction; restart server to invalidate."

- **Cache key must include `model` and `prompt_version` to survive prompt iteration.** Current spec keys by SOC alone. A hackathon reality: @fp-copywriter ships v1 prompt, gets feedback, ships v2. Builds spawned under v1 carry v1 description; the next sparkle for the same SOC under v2 should regenerate. With SOC-only key, students see v1 forever (until restart) and v2 students see v1 (cache hit). **Impact:** Real — the prompt **will** iterate. **Recommendation:** Either (a) version the prompt as a module constant `_PROMPT_VERSION = "v1"` and key by `(soc, _PROMPT_VERSION)`, or (b) explicitly accept "restart on prompt change" in Decision 4 with a comment in `_SYSTEM_PROMPT`. Pick (a) — costs nothing, prevents a footgun. Locale segmentation is correctly out of scope per Decision 4 — but apply the same versioning logic for the eventual `(soc, locale, _PROMPT_VERSION)` migration.

- **Cache concurrency: simultaneous misses for the same SOC fire N parallel Gemma calls.** The spec describes the cache as "the rate-limit guard," but a `dict` check-then-set under asyncio is **not** atomic across `await` boundaries. First request misses, awaits Gemma; second request misses (because first hasn't returned yet), awaits Gemma; both populate cache, both bill. Under cloud-demo concurrency (parents and students sparkle-clicking the same hot SOC), this can blow the OpenRouter rate limit on a popular career. **Impact:** Real for cloud-demo; latent for local Ollama. **Recommendation:** Replace the `dict[str, CareerDescription]` with `dict[str, asyncio.Future[CareerDescription]]` (or `asyncio.Task`). First miss creates and caches the Future; subsequent misses await the same Future. After completion, the Future stays in the cache (its `.result()` is the cached payload). This is a well-known idiom and adds ~10 LOC. Critical for cloud-demo correctness.

- **`validate_no_rpg_terms` does not exist as a public symbol in `pdf_copy.py` — only the frozenset does.** `pdf_copy.py` exports `RPG_TERMS_FORBIDDEN_IN_PDF` (frozenset) and `FORBIDDEN_IN_GEMMA_OUTPUT` (superset including stat abbreviations). `pdf_questions.py` independently defines `_FORBIDDEN_RE` and `_has_forbidden_term(text)` keyed off `FORBIDDEN_IN_GEMMA_OUTPUT`. Three codebases will validate Gemma output if this spec lands; the current count is two. **Impact:** Validator definition drift across modules. **Recommendation:** **Promote `_FORBIDDEN_RE` and `_has_forbidden_term` from `pdf_questions.py` into `pdf_copy.py` as public symbols** (`forbidden_terms_re()` factory taking a frozenset; `contains_forbidden_term(text, terms) -> bool`). Update `pdf_questions.py` to import from there. The new `career_description.py` then uses the same public helper, parameterized to `RPG_TERMS_FORBIDDEN_IN_PDF` (career descriptions can validly mention "earnings" but not the stat shorthand "ERN" — same ruleset as the rendered PDF text, **not** the stricter Gemma-output ruleset). A separate `voice_validation.py` is over-engineering for a hackathon; co-locating with the existing `pdf_copy.py` voice authority is the right move. Reject the spec's "(Possibly) export" hedging — it must be exported, not "possibly."

- **Decision 7 says "voice validation with one retry then `CareerDescriptionUnavailable`" but doesn't specify retry prompt strategy.** A blind retry with the same system prompt on the same SOC almost always returns the same drift. **Impact:** Wasted second Gemma call. **Recommendation:** §4 must specify: parse-failure retry uses the original prompt + a one-line user reminder ("Output **only** valid JSON matching the schema."); voice-failure retry includes the offending term(s) in a reminder ("Do not use the words: <list>."). Match the pattern in `boss_fights.py` if one exists; otherwise spell it out as 2 lines of pseudocode in the service signature.

- **Build-spawn parallelism: the eager 8s wall plus Gemma fanout's existing latency may exceed total spawn budget.** `_gemma_fanout` already runs ~5 narrative tasks + 3 Gemma calls in parallel via `asyncio.gather`. Adding a 6th independent Gemma call (the description) shares the semaphore (`gemma_client._get_semaphore()`) and competes for OpenRouter rate-limit headroom. Under cloud demo, p95 spawn latency is already the user-visible bottleneck. **Impact:** Could push spawn latency over the perceptual cliff on cold cache. **Recommendation:** Add the description task to the **same** `asyncio.gather` inside `_gemma_fanout` with `return_exceptions=True` rather than a separate `wait_for(...)` block. The 8s budget collapses into the existing fanout's effective wait (which is already bounded by the slowest individual call). One `gather`, one budget, one logging hook.

- **No path is out of scope (project rule): consider degenerate SOCs.** What happens when `get_task_breakdown` returns `top_5_activities=[]` or null? The spec defers this to @fp-data-reviewer (P1 test in §4) but the architecture must answer **before** review: does the service raise `CareerDescriptionUnavailable` (PDF section omitted), or fall through to `top_human_activities`, or generate a generic prompt without anchored activities? **Impact:** A SOC with no O*NET coverage will hit this path in production — Jeff's wife teaches deaf education, and the long tail of obscure SOCs is unevenly covered. **Recommendation:** §4 must specify the degrade-vs-fail policy. Recommended: prefer `top_human_activities` if `top_5_activities` is empty; if both empty, raise `CareerDescriptionUnavailable` immediately (before calling Gemma — saves a token spend on a SOC we cannot anchor). Document in Decision Log.

- **PDF section placement vs. existing page-1 layout.** §3 says "between header strip and pentagon" but `_build_page1` actually renders: profile/header strip → residency hint → data-coverage caveat → 0.75pt rule → **verdict line** → pentagon. The new section will displace the verdict line (which today is the immediate "what is this build worth" cue) or sit above it. **Impact:** Real — the verdict line is the first synthesizing sentence the reader sees; placing a 6-bullet description above it inverts the existing reading priority. This is a §3 design call, but the architecture must flag it: the exact insertion point is ambiguous in §3 and that ambiguity will surface in implementation. **Recommendation:** §3 must specify exact insertion: my recommendation is "after verdict line, before pentagon" (verdict still wins the eye, then context, then quantitative). Defer to @fp-design-visionary but require a one-line clarification in §3.

#### Degenerate-Case Recommendation (Question 3)

**Recommendation: tiered degrade with a hard floor — option (a). Reject the binary "raise on null" path.**

The current §4 design (raise `CareerDescriptionUnavailable` on any anchor failure) — and the architect's proposed strengthening at §5 item 7 (raise without invoking Gemma when both lists are empty) — is the wrong contract for this data. It would silently delete the "About this career" section from PDFs of paramedics, cardiologists, and financial analysts. A parent reading the PDF for "EMT at Indian Hills Community College" would see a pentagon and a verdict line but never learn what the job is — which is the exact failure mode this spec exists to fix.

Concrete tiered fallback inside `career_description.get_or_generate(soc_code, occupation_title)`:

1. **Tier A — full activity anchor.** `top_5_activities` non-null AND parseable JSON AND ≥4 entries. Use as today: prompt Gemma with `{occupation_title, description, parsed activities}`. Expected to cover ~93% of unique SOCs, ~93% of program-career path rows.
2. **Tier B — description-only anchor.** `top_5_activities` is NULL but the SOC has a row in `consumable.onet_work_profiles` (i.e., `description` is present). Prompt Gemma with `{occupation_title, description}` and an explicit instruction "infer 4–6 day-to-day tasks from the BLS occupation description; do not invent specifics not implied by the description." Mark `anchor_tier="description_only"`. Covers the 24 partial-tier SOCs.
3. **Tier C — title-only anchor.** SOC is not present in `consumable.onet_work_profiles` at all (the additional ~33 SOCs from the 57-SOC `program_career_paths` set). Prompt Gemma with `{occupation_title, soc_major_group_name}` and an explicit "this occupation is loosely defined; describe the typical day in plain terms; do not fabricate specific tools, software, or certifications." Mark `anchor_tier="title_only"`.
4. **Tier D — raise.** Only if SOC is malformed (fails `^\d{2}-\d{4}$`) or all three pulls fail (mcp_client unreachable). Then `CareerDescriptionUnavailable` is correct.

The tier label must be displayed in the UI for B and C — small confidence chip in the sparkle panel header card; one italicized disclaimer line in the PDF section. This is the project's bright line on observed-vs-AI-estimated data. Description-only and title-only generations are AI-inferred, not transcribed from O*NET activity importance scores, and the student/parent should know.

This recommendation is consistent with the project's `feedback_no_substitution_caveat` memo (no "Limited data" warnings on cards from CIP substitution) — that rule is about CIP→SOC crosswalk fallback at the *card* level. The disclaimer here is about *the description's anchor source*, which is a different concern: the user is being shown synthesized text and deserves to know whether it sits on activity-level evidence or on an occupation-summary blurb.

#### Tool Choice (Questions 4 and 5)

#### Multi-Detail and Catchall Risk (Question 6)

`onet_work_profiles.multi_detail_flag = True` for **76 of 798 rows** (9.5%). These are SOC major/minor groups that aggregate multiple detailed O*NET-SOC codes. For these rows, `top_5_activities` is computed across the group, not from a single occupation profile.

The 57-SOC NULL list above is dominated by `xx-xx99` "all, other" residual codes (25-1199, 19-1099, 25-3099, 27-1019, 25-1069, 11-9039, etc.). Those are not just multi-detail — they are *catchall* aggregations not present in `onet_work_profiles` at all. They will hit Tier C (title-only) under the recommendation above. Their `occupation_title` literally ends in "all other" or "all, other" — the prompt should be tuned to handle that text gracefully (the copywriter's prompt should not produce "Postsecondary teachers, all other professionals work in...").

Recommendations:

- Surface `multi_detail_flag` to the prompt for Tier A. When True, instruct Gemma "this code covers multiple related occupations; describe the day-to-day at the group level rather than picking specifics."
- Detect "all other" / "all, other" suffix in the occupation title (Tier C path) and have the copywriter author a dedicated catchall prompt variant.
- Do not raise on multi-detail rows. They have full activity data and should hit Tier A normally.

##### 🟡 Moderate

**M1. Eager fanout has no wall-clock budget; gemma_client transport timeout is 180s.**

**Impact:** A hung/stalled inference backend (Ollama deadlock, OpenRouter slow-loris) blocks `POST /builds` for up to ~180s before httpx times out. **Verdict:** Decision row 15 explicitly says "joining the existing gather; failure is non-fatal" and waives a separate budget. The implementation matches that revised decision. **M2. `_resolve_future` catches `BaseException`, swallowing `CancelledError` semantics.**

**Impact:** If the event loop is shutting down or the parent task is force-cancelled mid-generation, the `except BaseException` branch in `_resolve_future` (`career_description.py:277-279`) sets the cancellation as a regular exception on the future and the task ends cleanly. Coroutines awaiting that future will see `CancelledError` raised, which is correct — but the `_resolve_future` task itself does not re-raise, which is anti-pattern. In practice this only matters during interpreter shutdown.

**Location:** `backend/app/services/career_description.py:277-279`.

**Fix (optional, low priority):**
```python
except asyncio.CancelledError:
    _cache.pop(key, None)
    future.cancel()
    raise
except BaseException as exc:
    _cache.pop(key, None)
    future.set_exception(exc)
    raise
```

The current code has a `pragma: no cover - defensive` comment indicating awareness; leaving as-is is acceptable for hackathon scope.

**M3. Failed-future eviction defeats rate-limit guard under sustained Gemma outage.**

**Impact:** When `_resolve_future` evicts the cache entry on failure (`career_description.py:275, 278`), the next concurrent caller starts a fresh 2-call retry sequence. Under a sustained outage, every sparkle click on the same hot SOC burns 2 OpenRouter calls — the opposite of what a single-flight cache is supposed to do. This is the exact failure mode decision row 13 lists as a motivation ("blow OpenRouter rate limits").

**Mitigation already in place:** `_PROMPT_VERSION` guards drift but does not gate failure-retry. A short negative-cache TTL (e.g., 60s) on failures would close the gap.

**Recommendation:** Acceptable for hackathon; add a follow-up for the demo-scale guard. Not required for ship.

**M4. Frontend cache is keyed on SOC alone; backend keys on `(soc, prompt_version)`.**

**Impact:** A backend `_PROMPT_VERSION` bump (decision row 13's stated invalidation strategy) will still serve stale text in any tab that already has a resolved value cached client-side, until reload. Mostly cosmetic; tabs will pick up the fresh copy on the next page load. Worth a one-line comment on `careerDescriptionStore.ts:17` so the next maintainer doesn't trip on it.

##### 🔵 Minor

**B1. PDF defensive `_safe()` covers XML escaping but not embedded markdown fences leaking from Gemma into `summary`/`tasks`.**

If a Gemma response slips backticks INSIDE the summary (not at start/end where `_CODE_FENCE_RE` strips them), they survive parse + voice validation and render as literal backticks in the PDF. Aesthetic, not a security issue. Out of scope for this round.

**B2. Lazy PDF path has a third `except Exception` swallow that's broader than the spec.**
