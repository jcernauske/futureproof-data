## Method

Scenario-based chaos on an in-memory copy of the test fixture. Each scenario:

1. Mutates the raw TSV (or the post-ingestor flat-dict list).
2. Runs the real `OnetExperienceIngestor.flatten()` — this exercises the
   ingestor's required-field drop / type coercion logic.
3. Materializes the result as `raw.onet_experience` in in-memory DuckDB.
4. Executes every rule SQL in the opaque rules JSON and records PASS/FAIL.
5. Compares the scenario's FAIL set against a baseline FAIL set (same rules
   run against the unmutated fixture). The delta — **new fails the
   mutation introduced** — is the scenario's attribution.

Cycles use escalating rates (5%, 6%, 7%, 8%, 10%) over overlapping scenario
packs so each corruption is probed at multiple intensities.

Scenarios 7 (truncated ZIP / empty file) and 8 (wrong file in ZIP) hit the
ingestor directly — these are behavior checks, not DQ rule probes.
